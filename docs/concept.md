# Clearance

## The Problem

Traditional PR review was designed for a world where every line of code was written by a human. Reviewers would laboriously read every diff looking for bugs, test gaps, coding convention violations, and architectural issues. The process was painful but it made sense: a human wrote every line, so a human needed to read every line.

That world is ending. In the age of agentic AI:

- Much code is generated by agents, not typed line by line
- Automated reviewers already cover test coverage, architecture, and conventions
- Better guardrails — linting, CI, static analysis — catch the mechanical issues automatically
- The volume of PRs is rising as agents ship faster and faster

The result: the diff-browsing UI that GitHub shipped in 2008 is now actively working against you. You open a PR and feel a sense of dread. 400 lines of diff. Where do you even start?

**The diff is the wrong primary interface.** The review process is broken, and we need to rethink it from scratch.

---

## The Product

A desktop-native, chat-first tool for reviewing pull requests in the agentic era. Built natively on GitHub initially, with other platforms to follow.

### The Fundamental Inversion

Current tools say: *here's everything that changed, good luck.*

This tool says: *here's what matters — ask me about the rest.*

You open a PR and instead of a wall of diffs, an agent briefs you. It has already read the diff, the PR description, the linked ticket, and your review instructions. It opens with something like:

> *"This PR is doing one thing: migrating the order service to the new event bus. The author made three structural decisions you'll want to have opinions on. One thing surprised me — they touched the auth layer, which isn't mentioned in the ticket."*

You start from understanding, not from code. That alone removes most of the dread.

## Features

### The Opening Briefing

Every PR opens with a spoken-style briefing: what this PR is actually doing in plain language, what you're likely to care about given your history and instructions, and anything that surprised the agent. No scrolling, no orientation anxiety — you're grounded before you ask your first question.

### Guided Drill-Down

Review is conversational. You ask; the agent answers with the relevant diff surfaced inline:

- *"Show me the happy path"*
- *"What changed in the data model?"*
- *"How are they handling failures?"*
- *"Is this backwards compatible?"*
- *"Show me what's new vs. what's been moved"*

The diff is evidence. You pull it on demand; it doesn't get pushed at you. You never read a line you didn't ask to see.

### The Surprise Detector

Before you ask anything, the agent flags anomalies — the things that break the pattern:

- A new dependency that hasn't appeared in this repo before
- A function 3× longer than anything else in the module
- Tests deleted, not just added
- The PR description says "minor refactor" but 800 lines changed
- The auth layer was touched but this is a billing ticket

This is where most real problems hide. Not in the lines themselves, but in the things that don't fit.

### Review Chapters

Large PRs are one of the most common sources of reviewer friction. The traditional response is to ask the developer to break it up — which creates delay, back-and-forth, and usually doesn't happen anyway.

Clearance takes a different approach: the agent does the decomposition work for you, without requiring the developer to re-submit anything.

When it detects a large or mixed-concern diff, it carves the PR into logical **chapters** — coherent slices of intent that can be reviewed independently:

> *"This PR is doing three distinct things. I'd suggest reviewing them as:*
> *1. The data model changes (migrations + domain layer) — foundational, review first*
> *2. The new API endpoints — depends on 1*
> *3. The frontend integration — depends on 2, mostly mechanical"*

You step through the chapters at your own pace. Each one has its own briefing, its own thread of conversation, its own completion state. A 900-line PR stops feeling like a 900-line PR.

The agent also flags when chapters look genuinely independent — changes that have no coupling and could have been separate PRs. At that point you have a choice: review them as chapters and approve the whole thing, or ask the agent to draft a comment requesting the PR be split. The decision is yours; the analysis is already done.

This reframes the "please break up your PR" conversation entirely. It's no longer a gate you impose on the developer before review can start — it's something that happens inside the tool, transparently, in service of your understanding.

### Does This Match the Ticket?

One click. The agent pulls the linked issue, reads the acceptance criteria, and tells you honestly whether the PR delivers it — flagging missing requirements and scope creep. You stop having to hold the spec in your head while reading code.

### Ghost Commenting

You describe the comment; the agent writes it in your voice, in the appropriate tone for this team:

> *"Tell them to use the existing retry utility rather than rolling their own"*

Agent drafts:

> *"We've got `retryWithBackoff` in `src/utils/http.ts` that handles this — worth using here rather than a custom implementation so we keep retry behaviour consistent?"*

You click send, or tweak it. You stop writing diplomatic prose from scratch. You stop worrying about tone. You direct; the agent writes.

### The Pushback Assistant

When you need to request significant changes — or push something back entirely — the agent helps you articulate *why* in a way that's constructive and educational:

> *"Push this back — the spec should come first, the implementation is too far along"*

The agent drafts a comment that explains the concern clearly, suggests what a spec should cover, and frames it as guidance rather than rejection. Junior developers get a thoughtful explanation; you don't have to write it.

### The Completion Feeling

Right now you approve PRs with a background anxiety that you missed something. This tool replaces that with a genuine sense of completeness — because as you have the conversation, a checklist fills itself:

- Intent matches ticket ✓
- Critical paths covered ✓
- No unexplained new dependencies ✓
- Architectural concerns addressed ✓
- Error handling present ✓

When you approve, you know what you checked. The approval is a record of a real conversation, not a rubber stamp.

### Brief Me in 2 Minutes Mode

For PRs you need to bless but not deep-dive — a well-tested refactor, a dependency bump, a junior's straightforward feature — the agent gives you a timed, structured briefing. High confidence, low time. You approve knowing you got the essentials.

### Learns What You Care About

After enough reviews, the tool knows you:

- You always ask about error handling in the payments service → it front-loads that
- You never care about CSS changes → it deprioritises them
- You always push back on missing observability → it flags it proactively

The briefing gets better over time. It stops being generic and starts feeling like your own reviewer.

## Setup and Instructions

Each project links one-to-one with a repository and carries two layers of instructions:

1. **Global review instructions** — your personal preferences across all repos: tone, focus areas, what you always care about, what you never want to see surfaced
2. **Project-level instructions** — specific to this repo: architectural constraints, domain context, team norms, known sensitive areas

These work like CLAUDE.md files in Claude Code. Whether project-level instructions live in the tool or are committed to the repo itself is an open design question — committing them to the repo means the whole team benefits and they evolve with the codebase.

---

## Credential and Model Model

Initially built on Claude Code: users authenticate with their existing credentials and select their model. No new auth infrastructure. The tool is your own agent, running with your permissions, posting comments as you.

## Design Principles

- **Chat-first, diff-second.** The diff is evidence the agent can show you. It is not the interface.
- **You never see what you didn't ask for.** 400 lines of test boilerplate exist — you just don't have to read them.
- **Agent does the writing.** You direct; the agent composes. Comments, pushbacks, questions — all drafted for your approval.
- **Completion over coverage.** The goal isn't to have read every line. It's to genuinely understand the PR and know what you checked.
- **Desktop native.** Focused, fast, keyboard-driven. Not a web app bolted onto GitHub.

---

## Why Now

- Agentic PRs are getting longer and more numerous — cognitive load is growing, not shrinking
- Automated reviewers already handle the mechanical layer; humans should focus on intent, architecture, and correctness at a higher level
- The chat paradigm is now familiar enough that a review UI built around conversation won't feel alien
- Nobody has done this yet — existing tools still bolt AI commentary onto the 2008 diff UI

---

## Open Questions

- Should project-level review instructions be committed to the repo (like CLAUDE.md) or owned by the tool?
- Trust model: can the agent post comments autonomously, or always with human confirmation first?
- Team mode: shared global instructions for an engineering organisation vs. purely personal preferences?
- Multi-platform: GitLab, Bitbucket, Azure DevOps after GitHub?
- What's the right name? "PR reviewer" undersells it. This is closer to having a senior engineer who read everything and is ready to brief you.
